# WorkTime Sync — сценарий питча

5-7 минут. Перед началом — `docker compose up -d`. Демо-данные засеиваются
автоматически при первом старте (bootstrap).

## Подготовка демо

```bash
cp .env.example .env
# заполнить APP_ENCRYPTION_KEY и JWT_SECRET:
#   openssl rand -base64 32 → APP_ENCRYPTION_KEY
#   openssl rand -base64 64 → JWT_SECRET

# (опционально для AI) — GIGACHAT_CLIENT_ID + GIGACHAT_CLIENT_SECRET
# CA-сертификат Минцифры уже зашит в бинарь, отдельно класть не нужно.

docker compose up -d
# bootstrap создаст admin + 8 сотрудников + 3 команды + ~350 событий
```

Открыть [http://localhost:3000](http://localhost:3000) — Caddy проксирует на SvelteKit + Fiber API.

## Сценарий 1 — Employee (Анна, 3 минуты)

1. **Логин** anna@worktime.local / qwerty12345 → попадаем на dashboard
2. **Profile** — показываем рабочий график (10:00–19:00 МСК), добавляем отпуск через `/profile`:
   - Тип: Отпуск
   - Дата: +7 дней — +14 дней
   - Жмём «Добавить» → запись появляется в списке исключений
3. **Workload** — мои метрики (свежий профиль → A ≈ 0.97, R ≈ 0.10)
4. **Notifications** — статус подключения SSE (live)

**Что подсветить:**
- ✓ Все правки сразу видны в personal heatmap'е
- ✓ Подключение SSE — real-time канал, без поллинга
- ✓ Кнопка «Подтвердить актуальность» — отдельная от редактирования

## Сценарий 4 — PM (Мария, 1.5 минуты)

1. **Scheduler** — выбираем команду Distributed (TZ разные), длительность 60 мин, горизонт 7 дней
2. Жмём «Найти окна» → топ-3 слота с процентом доступности

**Что подсветить:**
- ✓ Слоты учитывают рабочие часы **каждого** участника (включая разные TZ)
- ✓ Исключения (Сергей на больничном) — учтены автоматически
- ✓ Если 100% невозможно — система показывает лучшее доступное

## Сценарий 5 — Admin + AI (1 минута)

1. **Admin → Rules** — показываем веса `w1..w5`, объясняем формулу R
2. Меняем `w1` с 0.30 на 0.35 → «Сохранить» → веса обновлены
3. **AI-чат** — задаём вопрос: «Почему у Сергея график считается неактуальным?»
   - Если GigaChat подключён → объяснимый ответ со ссылками на метрики
   - Если нет → rule-based fallback с теми же фактами

**Что подсветить:**
- ✓ Конфигурируемая формула риска — не зашита в код
- ✓ AI-ответ всегда основан на реальных данных (`recommender.md`), а не выдуман

## Что показать жюри коротко

1. **Реальные интеграции:** iCal, CalDAV (включая Yandex Календарь), Jira, Yandex Tracker
2. **6 метрик + интегральный риск R** — конфигурируемая формула, кэш в Redis
3. **15+ страниц UI** — все по дизайн-прототипу 1:1
4. **SSE-стрим** — реалтайм-уведомления без WebSocket-сложностей
5. **AI-модуль** — GigaChat с обязательным rule-based fallback
6. **Smart-notifier** — определяет получателей (сотрудник + manager + HR) и дедуп за 24ч
7. **OpenAPI 3.1 + Swagger UI** на `/swagger`

## План B (если демо рушится)

- Если БД пустая — `docker compose down -v && docker compose up -d` (volume снесётся, bootstrap пере-засеет)
- Если GigaChat недоступен — UI покажет badge "rule-based", функциональность не пропадает
- Если SSE не подключился — данные обновятся при `notifications.reload()` (вручную)
- Запись скринкаста — в `tmp/demo.mp4` (записать заранее)
