# Smart notifier — кому, что и когда отправлять

На входе JSON-список «кандидатов на уведомление» от backend'a:

```json
{
  "candidates": [
    {
      "user_id": "...",
      "role": "manager",
      "kind": "stale_profile",
      "subject_id": "emp-uuid",
      "subject_name": "Иван Иванов",
      "metrics_snapshot": { "A": 0.34, "R": 0.71 },
      "user_local_now": "2026-05-15T09:12:00+03:00",
      "last_notification_sent_at": "2026-05-13T08:00:00+03:00"
    }
  ],
  "limits": { "max_per_user_per_day": 3, "quiet_hours": ["20:00", "08:00"] }
}
```

## Задача

Выбрать подмножество уведомлений, которые действительно стоит отправить **сейчас**, и
сгенерировать персональный текст. Не отправлять всем одно и то же.

## Выход (только валидный JSON)

```json
{
  "to_send": [
    {
      "user_id": "...",
      "kind": "stale_profile",
      "title": "Иван Иванов давно не обновлял график",
      "body": "Профиль не обновлялся 47 дней. Это снижает точность планирования встреч команды.",
      "link": "/employees/emp-uuid",
      "reason": "high R (0.71) + within working hours + no notif in 48h"
    }
  ],
  "skipped": [
    { "user_id": "...", "reason": "quiet hours" }
  ]
}
```

## Правила

1. Не отправляй уведомления вне рабочих часов получателя (`quiet_hours`).
2. Не превышай `max_per_user_per_day`.
3. Сообщения **на русском**, без шаблонов «уведомляем вас о том, что». Прямо, по делу, до 140 символов в `body`.
4. `reason` — краткое объяснение для логов (не показывается пользователю).
5. Если кандидат критичный (`R > 0.85`) и до сих пор не уведомляли — отправляй даже в начале рабочего дня.
6. Если те же `kind+subject` уже отправляли менее 24 часов назад — пропусти, кроме `critical`.

Только валидный JSON, никакого markdown.
